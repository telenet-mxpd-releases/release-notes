01

The Commit Format

Every commit that should appear in the release notes follows one of two patterns. Copy the one you need and fill in the blanks.

Standard commit
[type] TICKET Description

[fix] ST-25829 Incorrect status after SLA breach

Commit requiring action after deploy
[type!] TICKET Description — steps

[data!] INCHMGPLTM-5274 Rename field — run SUB_MigrateOrgName

Rules
  • The type tag and its brackets are required: [feat], not feat: or feat.
  • A ticket reference is recommended for all types except [chore]. Untagged commits still appear in the release notes — under an Untagged warning section — but no ticket is needed.
  • The ! goes inside the brackets: [data!]. Implementation steps follow a space-em-dash-space: — .
  • Any commit containing [skip release-notes] is ignored by the pipeline entirely. The service uses this marker on its own automated commits to avoid infinite loops; you should not need it.
Ticket formats Two formats are accepted: Jira IDs (INCHMGPLTM-5273) and Clevr support IDs (ST-25829). Both appear as plain text in the release notes — no hyperlinks are generated automatically.
02

Commit Types

The type tag determines which section of the release notes receives your commit. Use the one that describes what the commit actually does.

[feat] Features Recommended [feat] INCHMGPLTM-5273 Add Priority field to Request
[fix] Bug Fixes Recommended [fix] ST-25829 Incorrect status after SLA breach
[data] Data Changes Recommended [data!] INCHMGPLTM-5280 Rename Organisation to Department — run SUB_MigrateOrgName
[config] Configuration Optional [config] Update default SLA thresholds for Priority 1
[refactor] Refactoring Optional [refactor] INCHMGPLTM-5261 Consolidate notification microflows
[chore] Maintenance Not required [chore] Remove unused microflows in Portal module
Data Changes get extra attention The Data Changes section is reviewed first by operations before every deployment. If your commit touches the database schema or content, use [data] — and add the ! flag with exact conversion microflow instructions if a manual step is needed.
03

Actions Required

The ! flag on any commit type adds an entry to the ⚠ Actions Required section at the top of the release notes — the first thing operations reads before deploying. Use it whenever someone needs to do something manually before or after the deployment goes live.

What to write after the separator

The text after — is copied verbatim into Actions Required. Write it as a direct instruction, not a description of what was done:

Don't — describes, doesn't instruct
[data!] INCHMGPLTM-5280 Rename field
— a migration is needed
Do — tells operations exactly what to run
[data!] INCHMGPLTM-5280 Rename field
— run SUB_MigrateOrgName after deploy
Common use cases
  • [data!] — a conversion microflow must be executed after deployment
  • [config!] — a constant or application setting must be changed in the cloud portal
  • [feat!] — a role permission must be granted before the feature is accessible
04

Sprint Lifecycle

The release notes collect all commits pushed to main after the sprint boundary was set. Starting a new sprint takes one commit.

1
Bump the sprint number
At the start of each sprint, one designated person edits CURRENT_RELEASE.txt in the root of the Mendix repo and sets it to the new sprint identifier — for example M26.07. Commit and push this change to main. Every commit after this point belongs to the new sprint.
2
Develop as normal
Push commits to main using the [type] format. The release notes regenerate automatically at midnight UTC each night — no action needed during the sprint.
3
Trigger manually when needed
Go to telenet-mxpd-releases/release-notes-service → Actions → Generate Release Notes → Run workflow. Use this before a deployment or after a batch of commits you want reflected immediately.
4
Share the link at sprint close
Copy the release notes URL and send it to operations and Telenet. The page stays live permanently — it is never deleted when a new sprint starts.
Sprint identifier format Use M<year>.<sprint> — for example M26.07 for the seventh sprint of 2026. The release notes URL replaces the dot with a hyphen: …/release-notes/M26-07.html.

What the page contains
  • An ⚠ Actions Required section at the top — present only if at least one ! commit exists in the sprint.
  • Sections for each commit type used: Features, Bug Fixes, Data Changes, Configuration, Refactoring, Maintenance.
  • Empty sections are hidden automatically.
  • An ⚠ Untagged section at the bottom if any commits did not match the format — a reminder to update those messages.
Sharing with Telenet The page is publicly accessible — no GitHub login required. Copy the URL directly into the sprint-close email. The link stays valid indefinitely.
06

FAQ

What happens if I forget the ticket reference?

The commit still appears in the release notes, in the correct section based on its type tag — the ticket field is simply left blank. Use this intentionally for [chore] commits where a ticket number doesn't apply.

What if I push a commit with no type tag at all?

It lands in an ⚠ Untagged section at the bottom of the page — flagged for review. Nothing is lost, but it signals that the commit message needs attention. You can amend the message and retrigger the workflow to move it into the right section.

Can I add a commit to the release notes after they've been published?

Yes. The release notes for a sprint are regenerated from scratch on every run, not appended to. Push the corrected commit, confirm that CURRENT_RELEASE.txt still contains the right sprint number, then trigger the workflow manually. The published page updates within a minute.

What is [skip release-notes] and should I ever use it?

The pipeline uses this marker on its own automated commits to avoid processing them in the next run. You should not need it. If you intentionally want a commit to be completely invisible to the release notes — a typo fix in an unrelated file, for example — you can add it to your commit message, but this should be rare.

The release notes are out of date. How do I refresh them immediately?

Go to telenet-mxpd-releases/release-notes-service → Actions → Generate Release Notes → Run workflow. This triggers an immediate run identical to the nightly scheduled run.

Two developers pushed commits at the same time. Is there a conflict risk?

The pipeline only reads commits from the Mendix repo; two developers pushing simultaneously does not cause a conflict there. The only write-back is the generated markdown file committed to the Mendix repo, and the HTML pushed to GitHub Pages — both are done by a single automated workflow run. If two runs ever overlap (unlikely given the midnight schedule), one will pull and retry once on conflict.
